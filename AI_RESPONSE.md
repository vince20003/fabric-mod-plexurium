1.  **Chunk Rendering:**

    *   **Frustum Culling:** Enhance existing frustum culling for chunks, entities, and particles to be more precise and efficient.
    *   **Block-Face Culling:** Aggressively cull unseen block faces during mesh generation (e.g., faces touching an opaque block, faces outside the current view frustum *if easily determinable without complex logic*).
    *   **Basic Occlusion Culling (Optional/Aggressive):** For opaque blocks, consider very basic software-based occlusion culling during meshing to avoid generating faces that are completely obscured by other opaque blocks within the same chunk section. *Keep this simple to avoid complexity.*
    *   **Entity Culling:** Implement custom, more aggressive frustum culling for entities. This might include culling entities that are entirely behind opaque blocks or too small to be visible from a distance (LOD consideration for culling, not model detail).
    *   **Vertical Render Distance Culling:** Introduce independent controls for vertical render distance (above and below the player) for both chunks and entities. This will be integrated into the culling logic during view frustum checks.

3.  **Lighting Optimization:**
    *   **Efficient Lighting Calculations:** While not replacing the entire lighting system, optimize how vanilla light values (block light, sky light) are sampled and applied during chunk meshing. This might involve caching light data where possible and reducing redundant calculations.
    *   **Shader Optimization:** Potentially optimize the default Minecraft shaders for lighting calculations, reducing instruction count or improving data locality. (This would be a careful, minimal change).

    *   **Vanilla Mesh Compatibility:** Design the meshing pipeline to generate meshes compatible with vanilla shaders.
    *   **Modded Block/Entity Compatibility:** Ensure that custom block models or entity renderers from other mods are handled gracefully. If a mod uses a custom block renderer, Plexurium should ideally still apply its chunk-level culling and GPU buffering to the chunk containing it efficiently.
    *   **Robust Error Handling:** Implement null checks, bounds checks, and safe concurrent access to avoid crashes.

    *   Provide a minimal set of hooks or events for other mod developers:
        *   Event/API for registering custom block/entity culling logic *if absolutely necessary* for very complex cases.
        *   Simple access to Plexurium's view frustum for custom rendering.
        *   Read-only access to some performance metrics.

By focusing on these points, Plexurium can achieve significant FPS gains by optimizing the bottlenecks in Minecraft's existing renderer without completely rewriting the engine, thus maintaining high compatibility and stability.
